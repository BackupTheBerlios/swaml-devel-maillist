Hi,

> My knowledge about SPARQL endpoints is very limited right now but as fas
> as I know it is a web service offering a SPARQL query service. Is it
> right? 

But yes, it's something similar, but using a XML format [1] instead SOAP
messages.

> Do you have in mind the technologies you are going to use to
> implement it? I suppose you just need to receive the web service query
> and use RDFLib SPARQL implementation to do the query. Then you need to
> return the results from RDFLib SPARQL in the format the web service will
> implement.

In server side the idea is to use the end-point that Sesame [2]
provides. Actually RDFLib hasn't any SPARQL client, therefore perhaps
we'll use PySesame [3] PySPARQL [4] to add to Buxon this feature. 

> In QualiPSo project next week we will go to Italy for a meeting and we
> will present there SWAML project as an example of the way we could
> follow to cover our information sources.

Excellent news. If you publish anything about this meeting related with
SWAML, please tell us to add it on SWAML-o-shere [5] (or feel free to
add it directly).

> Our main goal is the "Definition and prototyping of information
> management tools required for the collaborative platform" (think in CDE
> like GForge and in someway Trac) and thinking in a semantic
> collaborative platform seems to be a very interesting path to explore.

It sounds interesting... certainly GForge needs some love, and a
semantic point of view could help.
