> Yes, we are aware that there is a problem with memory comsumption. In
> fact, SWAML reads the whole mbox file into memory, which is clearly
> unacceptable for large mailing lists. We will explore some paths to
> avoid this. Probably it shouldn't be difficult to process the mbox file
> as a stream, and retaining in memory just the information required to
> wire the links between messages.
> 
> > I have seen also that you have added "apps/prototypes/sparql-endpoint".

> 
> Actually, we are working on this now. Our idea is to add to Buxon the
> feature of being able to query remote SPARQL endpoints (instead of
> fetching the RDF files). At the same time, we are trying to add an
> endpoint server to SWAML.

My knowledge about SPARQL endpoints is very limited right now but as fas
as I know it is a web service offering a SPARQL query service. Is it
right? Do you have in mind the technologies you are going to use to
implement it? I suppose you just need to receive the web service query
and use RDFLib SPARQL implementation to do the query. Then you need to
return the results from RDFLib SPARQL in the format the web service will
implement.

In QualiPSo project next week we will go to Italy for a meeting and we
will present there SWAML project as an example of the way we could
follow to cover our information sources.

We are also studying right now:
http://code.google.com/p/baetle/

to analyze our ontologies and reuse as much as possible other ontologies
that are already in use like SIOC, FOAF, DOAP and maybe Baetle.

Our main goal is the "Definition and prototyping of information
management tools required for the collaborative platform" (think in CDE
like GForge and in someway Trac) and thinking in a semantic
collaborative platform seems to be a very interesting path to explore.
